# Материалы проекта издательства

## 1. Первый оффер

От: Мария Крылова, выпускающий редактор издательства «Практика»

Тема: Практическое руководство по Claude Code для авторов учебников

Мы хотим заказать короткую, но плотную книгу для авторов и методистов, которые
пишут учебные пособия и хотят использовать Claude Code не как генератор текста,
а как рабочую среду для анализа материалов, планирования глав, проверки логики и
подготовки рукописи.

Нужен не рекламный текст и не обзор ИИ. Нужна книга, после которой автор сможет
открыть Claude Code, создать репозиторий проекта и провести учебник от замысла до
проверенной рукописи.

Предлагаем вести проект в нашем GitHub-профиле:

```text
edu-press/claude-code-textbook-guide
```

## 2. Что должно получиться

К концу проекта в репозитории должны быть:

- `BOOK_PLAN.md` — концепция книги, аудитория, результат обучения, структура.
- `chapters/` — черновики глав.
- `examples/` — практические примеры для читателя.
- `checks/` — контрольные списки качества.
- `MANUSCRIPT.md` — собранная рукопись для редактора.

Книга должна вести читателя от реальной проблемы к готовому решению:

1. Получен заказ или учебная задача.
2. Собраны материалы.
3. Описан идеальный результат.
4. Сформулированы главы и упражнения.
5. Проверена логика объяснения.
6. Подготовлена рукопись.

## 3. Требования редактора

- Не начинать с инструментов и команд.
- Сначала показывать проблему автора учебника.
- Все термины вводить до использования.
- Различать Claude, Claude Code, рабочий процесс, исполнителя и слеш-команду.
- Не писать «Claude всё сделает». Автор отвечает за цель и проверку.
- Каждый раздел должен заканчиваться практическим действием.

## 4. Требования методиста

Учебник должен добиваться умения, а не впечатления. Для каждой главы нужны:

- цель ученика;
- практическая ситуация;
- действие в Claude Code;
- типичная ошибка;
- мини-квиз;
- критерий понимания.

Особенно важно объяснить, что работа начинается с описания идеального результата.
Например: не «проанализировать материалы», а «получить план главы с целью,
примером, упражнением, проверкой и критерием готовности».

## 5. Технические замечания

Технический рецензент просит:

- показывать разницу между CLI и Desktop;
- подробно объяснить слеш-команды;
- держать все материалы в репозитории издательства;
- использовать GitHub issues для задач;
- фиксировать решения в `DECISIONS.md`;
- перед финальной сдачей проверить, что главы связаны одним сквозным примером.

## 6. Заметки автора

Нужно не потерять сюжет:

- издательство прислало оффер;
- автор принял заказ;
- создан репозиторий в профиле издательства;
- в репозитории появляются бриф, план, главы, проверки;
- Claude Code помогает переработать материалы в книгу;
- финальный результат — рукопись и пакет сдачи.

Возможный главный герой: Алексей, технический автор. Он хорошо знает предмет, но
хочет построить книгу как управляемый проект, а не как папку с разрозненными
черновиками.
